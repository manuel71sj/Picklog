# Picklog PRD

작성일: 2026-06-06
상태: Draft
대상 플랫폼: iOS, Android
목표: 개인용 생활정보 저장소 + 가벼운 수익화 가능성

## 1. 제품 요약

Picklog는 사용자가 사진, 쇼핑몰 링크, 유튜브/인스타그램 쇼츠 링크, 레시피 링크를 저장하면 AI가 자동으로 이름, 용도, 카테고리, 판매처, 가격, 출처, 핵심 메모를 추출해 개인 라이브러리로 정리하는 모바일 앱이다.

핵심 약속은 단순하다.

> 나중에 다시 찾고 싶은 생활 정보를 저장하면, 앱이 알아서 정리하고 필요할 때 바로 찾아준다.

첫 버전은 SNS 북마크, 쇼핑 위시리스트, 레시피 저장, 사진 메모가 흩어지는 문제를 하나의 개인 저장소로 합치는 데 집중한다. 협업, 공개 피드, 커뮤니티, 추천 커머스는 MVP 범위에서 제외한다.

## 2. 문제 정의

사용자는 사고 싶은 물건, 따라 해보고 싶은 레시피, 참고하고 싶은 쇼츠, 생활 팁, 선물 후보, 인테리어 아이디어를 여러 앱에 흩어져 저장한다.

현재 상태는 다음과 같다.

- 쇼핑몰 장바구니, 인스타 저장, 유튜브 재생목록, 브라우저 북마크, 카카오톡 나에게 보내기, 사진첩이 서로 분리되어 있다.
- 저장할 때는 쉽지만, 다시 찾을 때 "그게 어디 있었지?"가 된다.
- 링크 제목이나 썸네일만으로는 용도, 가격, 카테고리, 구매처, 왜 저장했는지 알기 어렵다.
- 시간이 지나면 링크가 사라지거나 가격이 바뀌고, 저장한 이유도 잊힌다.

## 3. 타깃 사용자

### 1차 사용자

개인적으로 쇼핑, 요리, 생활 팁, 콘텐츠 참고 자료를 많이 저장하는 모바일 사용자.

대표 페르소나:

- 20-40대 모바일 쇼핑 사용자
- 인스타그램, 유튜브 쇼츠, 블로그/레시피 링크를 자주 저장하는 사용자
- 선물, 인테리어, 육아, 요리, 여행, 취미 물품을 나중에 다시 비교하는 사용자
- Notion, Apple Notes, Google Keep, 카카오톡 나에게 보내기를 쓰지만 정리가 귀찮은 사용자

### 초기 웨지

첫 번째 유료 가치가 가장 강한 웨지는 **"AI 쇼핑/생활 링크 저장함"**이다.

이유:

- 사진/링크 입력 모두 자연스럽다.
- 가격, 판매처, 카테고리, 용도 같은 구조화 메타데이터의 가치가 명확하다.
- 무료 메모 앱과 다르게 "자동 정리"가 바로 체감된다.
- 이후 레시피, 쇼츠, 생활 팁으로 확장하기 쉽다.

## 4. 목표

### 제품 목표

- 사용자가 사진이나 링크를 3초 안에 저장할 수 있다.
- AI가 저장 항목의 메타데이터 초안을 자동 생성한다.
- 사용자가 카테고리, 판매처, 가격, 용도, 이름, 태그로 빠르게 검색할 수 있다.
- 불필요한 항목은 아카이브하거나 영구 삭제할 수 있다.
- 개인용 저장소로 신뢰할 수 있도록 원본 링크, 추출 근거, 수정 이력을 보존한다.

### 사업 목표

- 무료 사용자는 제한된 저장 개수와 AI 추출 횟수로 핵심 가치를 경험한다.
- 유료 사용자는 더 많은 저장, 고급 검색, 가격 추적, 백업/동기화, AI 재분류를 얻는다.
- 초기 수익화는 광고나 제휴보다 구독/일회성 Pro 잠금 해제를 우선한다.

## 5. 비목표

MVP에서 하지 않는다.

- 공개 피드, 팔로우, 공유 커뮤니티
- 앱 내 직접 결제형 커머스
- 자동 구매, 자동 예약, 대리 주문
- 모든 쇼핑몰 가격의 실시간 보장
- 인스타그램/유튜브 영상 다운로드
- 저작권 콘텐츠 복제 저장
- 복잡한 팀 협업, 가족 공유 저장소
- 크롤링 기반 대규모 가격 비교 서비스

## 6. 핵심 사용자 흐름

### 흐름 A: 사진 저장

1. 사용자가 앱에서 사진을 촬영하거나 사진첩 이미지를 선택한다.
2. 앱이 이미지를 업로드하고 AI 추출을 시작한다.
3. AI가 이름, 카테고리, 용도, 예상 가격, 브랜드/판매처 후보, 태그, 설명을 생성한다.
4. 사용자는 결과를 확인하고 필요한 필드를 수정한다.
5. 저장하면 항목이 라이브러리에 추가된다.

### 흐름 B: 링크 저장

1. 사용자가 공유 시트에서 Picklog를 선택하거나 앱에 링크를 붙여넣는다.
2. 앱이 URL 타입을 판별한다.
3. 쇼핑몰, 유튜브, 인스타그램, 레시피/블로그, 일반 웹페이지 중 하나로 분류한다.
4. 가능한 경우 공식 API, oEmbed, 페이지 메타태그, 사용자 제공 텍스트를 조합해 기본 메타데이터를 얻는다.
5. AI가 Picklog 표준 스키마로 구조화한다.
6. 사용자는 결과를 확인하고 저장한다.

### 흐름 C: 검색과 확인

1. 사용자가 검색창에 "선물", "10만원 이하", "에어프라이어", "저녁 레시피", "인스타 조명" 등을 입력한다.
2. 앱이 이름, 카테고리, 용도, 태그, 판매처, 가격, 원본 출처를 통합 검색한다.
3. 결과는 카드 리스트로 표시된다.
4. 사용자는 필터를 켜서 카테고리, 가격대, 출처, 저장일, 상태를 좁힌다.
5. 상세 화면에서 원본 링크 열기, 메모 수정, 아카이브, 삭제를 수행한다.

### 흐름 D: 정리

1. 사용자가 항목을 왼쪽으로 스와이프한다.
2. 빠른 액션으로 아카이브 또는 삭제를 선택한다.
3. 아카이브 항목은 기본 검색에서 제외되지만 별도 필터로 볼 수 있다.
4. 삭제 항목은 휴지통으로 이동하고 30일 후 자동 삭제된다.

## 7. 기능 요구사항

### 7.1 저장 입력

필수:

- 사진 촬영
- 사진첩에서 이미지 선택
- 앱 내 링크 붙여넣기
- iOS/Android 공유 시트에서 링크 또는 이미지 받기
- 저장 전 로딩, 실패, 재시도 상태 표시

후순위:

- 여러 사진 한번에 저장
- 웹 브라우저 확장
- 데스크톱 웹 앱

### 7.2 AI 메타데이터 추출

공통 필드:

- `title`: 사용자가 이해하기 쉬운 이름
- `summary`: 한두 문장 설명
- `category`: 표준 카테고리
- `use_case`: 사용 목적
- `tags`: 검색용 태그
- `source_type`: photo, shopping_url, youtube_url, instagram_url, recipe_url, article_url, unknown
- `source_url`: 원본 링크
- `thumbnail_url`: 썸네일 또는 업로드 이미지
- `confidence`: AI 추출 신뢰도
- `extraction_notes`: 불확실하거나 사용자가 확인해야 할 내용

쇼핑 필드:

- `seller`: 판매처
- `brand`: 브랜드
- `product_name`: 상품명
- `price`: 가격
- `currency`: 통화
- `price_observed_at`: 가격 확인 시각
- `purchase_intent`: 사고 싶음, 비교 중, 이미 구매, 선물 후보

레시피 필드:

- `dish_name`: 요리명
- `ingredients`: 주요 재료
- `cooking_time_minutes`: 조리 시간
- `difficulty`: 쉬움, 보통, 어려움
- `meal_type`: 아침, 점심, 저녁, 간식, 반찬, 디저트

쇼츠/영상 필드:

- `creator`: 채널 또는 계정명
- `platform`: YouTube, Instagram, TikTok, other
- `video_title`: 영상 제목
- `duration_seconds`: 가능할 때만 저장
- `why_saved`: 저장 이유 추정

### 7.3 검색

필수:

- 전체 텍스트 검색
- 카테고리 필터
- 출처 타입 필터
- 가격대 필터
- 판매처 필터
- 태그 필터
- 아카이브 포함/제외 토글
- 최근 저장순, 가격 낮은순, 이름순 정렬

후순위:

- 자연어 검색: "지난달에 저장한 5만원 이하 주방용품"
- 유사 이미지 검색
- 중복 항목 자동 묶기

### 7.4 상세 화면

필수:

- 대표 이미지/썸네일
- 제목, 요약, 카테고리, 태그
- 원본 링크 열기
- 가격/판매처/용도 표시
- 사용자 메모
- 모든 AI 필드 직접 수정
- AI 재추출
- 아카이브
- 삭제

### 7.5 아카이브와 삭제

필수:

- `active`, `archived`, `deleted` 상태
- 아카이브는 복원 가능
- 삭제는 휴지통으로 이동
- 휴지통에서 30일 내 복원 가능
- 30일 후 영구 삭제

### 7.6 계정과 동기화

MVP 권장:

- Apple 로그인
- Google 로그인
- 익명 로컬 모드
- 클라우드 동기화는 로그인 사용자에게 제공

로컬 모드 사용자는 기기 변경 시 데이터가 사라질 수 있음을 명확히 표시한다.

### 7.7 수익화

무료:

- 저장 항목 100개
- 월 AI 추출 30회
- 기본 검색
- 로컬 저장 또는 단일 기기 동기화

Pro 구독:

- 저장 항목 무제한 또는 높은 상한
- 월 AI 추출 확대
- 클라우드 백업/동기화
- 고급 필터
- AI 재분류
- 중복 감지
- 가격 변경 알림

일회성 Pro 잠금 해제 옵션:

- 개인 프로젝트 성격상 구독 저항이 크면 일회성 결제를 제공한다.
- 단, 지속적인 AI 비용이 드는 기능은 크레딧 또는 월 한도로 제한한다.

## 8. 권장 접근안

### Approach A: 로컬 우선 개인 저장소

요약:
기기 로컬 DB를 먼저 만들고, AI 추출만 서버에서 처리한다. 로그인 없이 바로 쓸 수 있다.

장점:

- 가장 빨리 만들 수 있다.
- 개인 프로젝트에 맞게 사용 마찰이 낮다.
- 초기 서버 비용이 낮다.

단점:

- 동기화와 백업이 약하다.
- 수익화 기능을 붙일 때 계정 전환이 필요하다.

### Approach B: 클라우드 우선 Pro 앱

요약:
처음부터 계정, 클라우드 저장, 구독, 서버 AI 파이프라인을 갖춘다.

장점:

- 수익화와 동기화가 깔끔하다.
- iOS/Android 간 경험이 일관된다.
- 가격 추적, AI 재분류 같은 Pro 기능 확장이 쉽다.

단점:

- 첫 출시가 늦어진다.
- 개인용 앱으로 쓰기 전에 가입 장벽이 생긴다.
- 운영 비용이 먼저 발생한다.

### Approach C: 저장 봇 + 모바일 뷰어

요약:
카카오톡/텔레그램/디스코드 같은 채널에 링크를 보내면 서버가 정리하고, 모바일 앱은 검색/확인 뷰어 역할을 한다.

장점:

- 저장 입력이 매우 빠르다.
- "나에게 보내기" 습관을 그대로 흡수할 수 있다.
- 향후 팀/가족 공유로 확장할 수 있다.

단점:

- iOS/Android 앱의 기본 경험이 약해진다.
- 외부 플랫폼 의존성이 커진다.
- 앱스토어용 독립 앱 가치가 흐려질 수 있다.

### 추천

MVP는 **Approach A: 로컬 우선 개인 저장소**로 시작한다.

단, 데이터 모델은 처음부터 계정/동기화/구독을 붙일 수 있게 설계한다. 첫 버전의 성공 기준은 "내가 실제로 매일 저장하고 다시 찾는다"이지, 완성형 SaaS 구조가 아니다.

## 11. AI 추출 설계

### 원칙

- AI 결과는 초안이며, 사용자가 언제든 수정할 수 있다.
- 신뢰도가 낮은 필드는 "확인 필요"로 표시한다.
- 가격은 자동 추출 시점의 관측값으로 표시하고, 실시간 최저가처럼 표현하지 않는다.
- 원본 링크와 추출 시점을 함께 저장한다.

### 파이프라인

1. 입력 수집: 이미지 또는 URL
2. URL 타입 판별
3. 메타데이터 사전 수집
   - YouTube: 공식 Data API로 가능한 메타데이터 수집
   - 일반 URL: oEmbed 또는 Open Graph 메타태그 사용
   - 쇼핑몰/레시피: 페이지 메타태그와 본문 일부 사용
   - Instagram: 공식/허용된 임베드 메타데이터가 없으면 사용자가 제공한 링크와 썸네일 수준으로 제한
4. AI 구조화
5. 스키마 검증
6. 사용자 확인 화면
7. 저장

### 스키마 버전

AI 출력은 `schema_version`을 둔다. 예: `picklog_item_v1`.

모델이 바뀌어도 기존 저장 데이터는 깨지지 않아야 한다.

## 12. 정책과 플랫폼 제약

### iOS

- 사용자의 사진, 링크, 메모는 개인 데이터이므로 제3자 AI 처리 사실과 목적을 명확히 고지해야 한다.
- Pro 기능은 디지털 기능 잠금 해제이므로 App Store 결제 정책에 맞춰 인앱 구매 또는 구독을 우선 고려한다.
- 사용자가 저장한 외부 콘텐츠가 앱 내에서 보이는 경우, 부적절한 콘텐츠 신고/삭제/차단 흐름을 최소한 갖춘다.

### Android

- AI가 생성하거나 요약한 내용에 대해 사용자 신고/피드백 수단을 제공한다.
- 구독은 지속적 가치를 제공해야 한다. 단순 일회성 AI 추출 묶음만 구독으로 파는 것은 피한다.

### 외부 플랫폼

- YouTube는 공식 API로 제목, 설명, 썸네일, 채널명 등 영상 리소스 메타데이터를 얻는 방향이 안전하다.
- oEmbed는 URL을 직접 파싱하지 않고 임베드 가능한 대표 정보와 썸네일을 가져오는 표준 수단으로 사용할 수 있다.
- Instagram, 쇼핑몰, 레시피 사이트는 플랫폼별 접근 제한이 다르므로 무리한 크롤링보다 사용자 제공 링크, 공개 메타태그, 공식 API, 수동 보정 UX를 조합한다.

## 18. 구현 권장안

- 앱: React Native + Expo

- 서버: Supabase 또는 Firebase

주의:

- 공유 시트 수신은 단순 공유보다 까다롭다. Expo의 기본 Sharing API는 "내 앱에서 외부로 공유"에는 적합하지만, "외부 앱에서 Picklog로 저장"은 iOS Share Extension/Android Share Intent 구현 검증이 필요하다.
- 따라서 Alpha에서는 앱 안의 링크 붙여넣기와 사진 선택을 먼저 완성하고, Beta에서 공유 시트를 붙인다.

주요 엔지니어링 이슈:

- 로컬 우선이어도 미래 동기화를 위해 모든 레코드에 충돌 해결용 필드를 준비해야 한다.

테스트 공백:

- URL 보안 유닛 테스트가 없다.
- 삭제 cascade 테스트가 없다.
- 30개 데이터셋 기반 AI eval 기준이 없다.
- 공유 시트는 Expo 문서상 네이티브 설정과 실험적 수신 흐름이 있으므로 실제 기기 E2E가 필요하다.

### 6. DX 리뷰

적용 범위: 제한적.

이 PRD는 개발자용 API/SDK/CLI 제품이 아니라 소비자 모바일 앱이다. 따라서 정식 DX 리뷰 대상은 아니다. 다만 개인 프로젝트의 개발자 경험 관점에서 다음은 PRD에 남길 가치가 있다.

- 로컬 개발에서 AI 서버 없이 수동 입력/fixture로 앱을 실행할 수 있어야 한다.
- 30개 Alpha 데이터셋은 앱 fixture와 AI eval fixture로 동시에 쓰여야 한다.
- 서버 AI API는 OpenAPI 또는 간단한 typed contract로 고정해 모바일/서버 병렬 개발을 가능하게 한다.

### 7. 결정 감사 로그

자동 결정:

- `Approach A` 유지: Alpha 개인 사용성과 비용 제어가 가장 중요하다.
- Pro 기능은 Alpha 필수에서 제외: 수익화 준비는 데이터 모델과 설정 화면 경계로 충분하다.
- DX 리뷰는 축소 적용: 이 제품은 개발자 플랫폼이 아니다.

사용자 확인 결정:

- PRD 수정/리뷰 방향: `3번` 균형안.

남은 사용자 결정:

- 첫 구현 범위에서 공유 시트를 Alpha에서 완전히 제외할지, 기술 spike만 할지.
- 클라우드 백업을 Beta 필수로 둘지, v1 전환 항목으로 늦출지.
- AI 추출 raw 로그 보존 기간을 0일, 7일, 30일 중 어디로 둘지.

### 8. 권장 PRD 패치 목록

다음 패치를 별도 작업으로 PRD 본문에 반영하는 것을 권장한다.

1. 7.1 저장 입력에서 공유 시트를 필수에서 제거하고 Beta 항목으로 일관화한다.

### 9. 완료 기준 제안

Alpha 완료:

- 본인이 30개 실제 항목을 저장한다.
- 저장 1건당 평균 완료 시간이 10초 이하이다.
- 30개 데이터셋에서 필수 필드 허용 정확도 80% 이상이다.

- AI 실패, 링크 접근 실패, 낮은 신뢰도 상태에서도 수동 저장을 완료할 수 있다.

Beta 완료:

- 공유 시트 수신이 iOS/Android 실제 기기에서 동작한다.
- 로그인/백업/복원이 최소 1회 이상 검증된다.

- Pro 제한 UI가 저장 흐름을 방해하지 않는다.
