# Picklog PRD

작성일: 2026-06-06
상태: MVP PRD v1
대상 플랫폼: iOS, Android
목표: 개인용 생활정보 저장소 + 가벼운 수익화 가능성

## 1. 제품 요약

Picklog는 사용자가 사진, 쇼핑몰 링크, 유튜브/인스타그램 쇼츠 링크, 레시피 링크를 저장하면 AI가 자동으로 이름, 용도, 카테고리, 판매처, 가격, 출처, 핵심 메모를 추출해 개인 라이브러리로 정리하는 모바일 앱이다.

핵심 약속은 단순하다.

> 나중에 다시 찾고 싶은 생활 정보를 저장하면, 앱이 알아서 정리하고 필요할 때 바로 찾아준다.

첫 버전은 SNS 북마크, 쇼핑 위시리스트, 레시피 저장, 사진 메모가 흩어지는 문제를 하나의 개인 저장소로 합치는 데 집중한다. 협업, 공개 피드, 커뮤니티, 추천 커머스는 MVP 범위에서 제외한다.

## 2. 문제 정의

사용자는 사고 싶은 물건, 따라 해보고 싶은 레시피, 참고하고 싶은 쇼츠, 생활 팁, 선물 후보, 인테리어 아이디어를 여러 앱에 흩어져 저장한다.

현재 상태는 다음과 같다.

- 쇼핑몰 장바구니, 인스타 저장, 유튜브 재생목록, 브라우저 북마크, 카카오톡 나에게 보내기, 사진첩이 서로 분리되어 있다.
- 저장할 때는 쉽지만, 다시 찾을 때 "그게 어디 있었지?"가 된다.
- 링크 제목이나 썸네일만으로는 용도, 가격, 카테고리, 구매처, 왜 저장했는지 알기 어렵다.
- 시간이 지나면 링크가 사라지거나 가격이 바뀌고, 저장한 이유도 잊힌다.

## 3. 타깃 사용자

### 1차 사용자

개인적으로 쇼핑, 요리, 생활 팁, 콘텐츠 참고 자료를 많이 저장하는 모바일 사용자.

대표 페르소나:

- 20-40대 모바일 쇼핑 사용자
- 인스타그램, 유튜브 쇼츠, 블로그/레시피 링크를 자주 저장하는 사용자
- 선물, 인테리어, 육아, 요리, 여행, 취미 물품을 나중에 다시 비교하는 사용자
- Notion, Apple Notes, Google Keep, 카카오톡 나에게 보내기를 쓰지만 정리가 귀찮은 사용자

### 초기 웨지

첫 번째 유료 가치가 가장 강한 웨지는 **"AI 쇼핑/생활 링크 저장함"**이다.

이유:

- 사진/링크 입력 모두 자연스럽다.
- 가격, 판매처, 카테고리, 용도 같은 구조화 메타데이터의 가치가 명확하다.
- 무료 메모 앱과 다르게 "자동 정리"가 바로 체감된다.
- 이후 레시피, 쇼츠, 생활 팁으로 확장하기 쉽다.

## 4. 목표

### 제품 목표

- 사용자가 사진이나 링크를 3초 안에 저장할 수 있다.
- AI가 저장 항목의 메타데이터 초안을 자동 생성한다.
- 사용자가 카테고리, 판매처, 가격, 용도, 이름, 태그로 빠르게 검색할 수 있다.
- 불필요한 항목은 아카이브하거나 영구 삭제할 수 있다.
- 개인용 저장소로 신뢰할 수 있도록 원본 링크, 추출 근거, 수정 이력을 보존한다.

### 사업 목표

- 무료 사용자는 제한된 저장 개수와 AI 추출 횟수로 핵심 가치를 경험한다.
- 유료 사용자는 더 많은 저장, 고급 검색, 가격 추적, 백업/동기화, AI 재분류를 얻는다.
- 초기 수익화는 광고나 제휴보다 구독/일회성 Pro 잠금 해제를 우선한다.

## 5. 비목표

MVP에서 하지 않는다.

- 공개 피드, 팔로우, 공유 커뮤니티
- 앱 내 직접 결제형 커머스
- 자동 구매, 자동 예약, 대리 주문
- 모든 쇼핑몰 가격의 실시간 보장
- 인스타그램/유튜브 영상 다운로드
- 저작권 콘텐츠 복제 저장
- 복잡한 팀 협업, 가족 공유 저장소
- 크롤링 기반 대규모 가격 비교 서비스

## 6. 핵심 사용자 흐름

### 흐름 A: 사진 저장

1. 사용자가 앱에서 사진을 촬영하거나 사진첩 이미지를 선택한다.
2. 앱이 이미지를 업로드하고 AI 추출을 시작한다.
3. AI가 이름, 카테고리, 용도, 예상 가격, 브랜드/판매처 후보, 태그, 설명을 생성한다.
4. 사용자는 결과를 확인하고 필요한 필드를 수정한다.
5. 저장하면 항목이 라이브러리에 추가된다.

### 흐름 B: 링크 저장

1. 사용자가 앱 안의 입력창에 링크를 붙여넣는다. 공유 시트 수신은 Beta에서 추가한다.
2. 앱이 URL 타입을 판별한다.
3. 쇼핑몰, 유튜브, 인스타그램, 레시피/블로그, 일반 웹페이지 중 하나로 분류한다.
4. 가능한 경우 공식 API, oEmbed, 페이지 메타태그, 사용자 제공 텍스트를 조합해 기본 메타데이터를 얻는다.
5. AI가 Picklog 표준 스키마로 구조화한다.
6. 사용자는 결과를 확인하고 저장한다.

### 흐름 C: 검색과 확인

1. 사용자가 검색창에 "선물", "10만원 이하", "에어프라이어", "저녁 레시피", "인스타 조명" 등을 입력한다.
2. 앱이 이름, 카테고리, 용도, 태그, 판매처, 가격, 원본 출처를 통합 검색한다.
3. 결과는 카드 리스트로 표시된다.
4. 사용자는 필터를 켜서 카테고리, 가격대, 출처, 저장일, 상태를 좁힌다.
5. 상세 화면에서 원본 링크 열기, 메모 수정, 아카이브, 삭제를 수행한다.

### 흐름 D: 정리

1. 사용자가 항목을 왼쪽으로 스와이프한다.
2. 빠른 액션으로 아카이브 또는 삭제를 선택한다.
3. 아카이브 항목은 기본 검색에서 제외되지만 별도 필터로 볼 수 있다.
4. 삭제 항목은 휴지통으로 이동하고 30일 후 자동 삭제된다.

## 7. 기능 요구사항

### 7.1 저장 입력

필수:

- 사진 촬영
- 사진첩에서 이미지 선택
- 앱 내 링크 붙여넣기
- 저장 전 로딩, 실패, 재시도 상태 표시
- AI 추출 실패 시 수동 입력으로 저장 완료

후순위:

- iOS/Android 공유 시트에서 링크 또는 이미지 받기
- 여러 사진 한번에 저장
- 웹 브라우저 확장
- 데스크톱 웹 앱

### 7.1.1 캡처 상태

저장 입력 화면은 다음 상태를 명확히 보여준다.

| 상태 | 사용자에게 보여줄 것 | 사용자가 할 수 있는 것 |
| --- | --- | --- |
| 입력 대기 | 사진 선택, 촬영, 링크 붙여넣기 | 입력 시작 |
| 추출 중 | 진행 표시, 취소 버튼 | 취소 |
| 낮은 신뢰도 | 확인 필요 필드 강조 | 수정 후 저장 |
| 일부 필드 실패 | 실패한 필드와 원인 | 수동 입력, 재시도 |
| 지원하지 않는 링크 | 링크 접근 제한 또는 플랫폼 제한 | 원본 링크만 저장 |
| AI 실패 | 추출 실패 메시지 | 수동 저장, 재시도 |
| 저장 완료 | 저장된 항목 요약 | 상세 보기, 계속 저장 |

### 7.2 AI 메타데이터 추출

공통 필드:

- `title`: 사용자가 이해하기 쉬운 이름
- `summary`: 한두 문장 설명
- `category`: 표준 카테고리
- `use_case`: 사용 목적
- `tags`: 검색용 태그
- `source_type`: photo, shopping_url, youtube_url, instagram_url, recipe_url, article_url, unknown
- `source_url`: 원본 링크
- `thumbnail_url`: 썸네일 또는 업로드 이미지
- `confidence`: AI 추출 신뢰도
- `extraction_notes`: 불확실하거나 사용자가 확인해야 할 내용

쇼핑 필드:

- `seller`: 판매처
- `brand`: 브랜드
- `product_name`: 상품명
- `price`: 가격
- `currency`: 통화
- `price_observed_at`: 가격 확인 시각
- `purchase_intent`: 사고 싶음, 비교 중, 이미 구매, 선물 후보

레시피 필드:

- `dish_name`: 요리명
- `ingredients`: 주요 재료
- `cooking_time_minutes`: 조리 시간
- `difficulty`: 쉬움, 보통, 어려움
- `meal_type`: 아침, 점심, 저녁, 간식, 반찬, 디저트

쇼츠/영상 필드:

- `creator`: 채널 또는 계정명
- `platform`: YouTube, Instagram, TikTok, other
- `video_title`: 영상 제목
- `duration_seconds`: 가능할 때만 저장
- `why_saved`: 저장 이유 추정

### 7.3 검색

필수:

- 전체 텍스트 검색
- 카테고리 필터
- 출처 타입 필터
- 가격대 필터
- 판매처 필터
- 태그 필터
- 아카이브 포함/제외 토글
- 최근 저장순, 가격 낮은순, 이름순 정렬

후순위:

- 자연어 검색: "지난달에 저장한 5만원 이하 주방용품"
- 유사 이미지 검색
- 중복 항목 자동 묶기

### 7.4 상세 화면

필수:

- 대표 이미지/썸네일
- 제목, 요약, 카테고리, 태그
- 원본 링크 열기
- 가격/판매처/용도 표시
- 사용자 메모
- 모든 AI 필드 직접 수정
- AI 추정값과 사용자 확정값 구분
- 낮은 신뢰도 필드의 확인 필요 표시
- AI 재추출
- 아카이브
- 삭제

### 7.5 아카이브와 삭제

필수:

- `active`, `archived`, `deleted` 상태
- 아카이브는 복원 가능
- 삭제는 휴지통으로 이동
- 휴지통에서 30일 내 복원 가능
- 30일 후 영구 삭제
- 영구 삭제 시 항목, 첨부, 추출 원본, 가격 기록이 보존 정책에 따라 삭제 또는 익명화

### 7.6 계정과 동기화

Alpha:

- 익명 로컬 모드
- 로그인 없이 저장, 검색, 아카이브, 삭제 가능
- 로컬 모드 사용자는 기기 변경 시 데이터가 사라질 수 있음을 명확히 표시

Beta:

- Apple 로그인
- Google 로그인
- 클라우드 백업/복원
- 로그인 사용자의 기기 간 동기화

동기화 준비:

- Alpha 데이터 모델에도 `local_id`, `remote_id`, `device_id`, `sync_state`, `version`, `tombstone_until`을 둔다.
- Alpha에서는 동기화 필드를 로컬에서만 사용하고, Beta에서 서버 동기화와 연결한다.
- 삭제 항목은 즉시 완전 제거하지 않고 `tombstone_until`까지 남겨 이후 동기화 충돌을 처리할 수 있게 한다.

### 7.7 수익화

무료:

- 저장 항목 100개
- 월 AI 추출 30회
- 기본 검색
- 로컬 저장

Pro 구독:

- 저장 항목 무제한 또는 높은 상한
- 월 AI 추출 확대
- 클라우드 백업/동기화
- 고급 필터
- AI 재분류
- 중복 감지
- 가격 변경 알림

일회성 Pro 잠금 해제 옵션:

- 개인 프로젝트 성격상 구독 저항이 크면 일회성 결제를 제공한다.
- 단, 지속적인 AI 비용이 드는 기능은 크레딧 또는 월 한도로 제한한다.

Alpha에서는 결제를 구현하지 않는다. 대신 설정 화면에 AI 사용량과 Pro 준비 영역만 둔다.

## 8. 권장 접근안

### Approach A: 로컬 우선 개인 저장소

요약:
기기 로컬 DB를 먼저 만들고, AI 추출만 서버에서 처리한다. 로그인 없이 바로 쓸 수 있다.

장점:

- 가장 빨리 만들 수 있다.
- 개인 프로젝트에 맞게 사용 마찰이 낮다.
- 초기 서버 비용이 낮다.

단점:

- 동기화와 백업이 약하다.
- 수익화 기능을 붙일 때 계정 전환이 필요하다.

### Approach B: 클라우드 우선 Pro 앱

요약:
처음부터 계정, 클라우드 저장, 구독, 서버 AI 파이프라인을 갖춘다.

장점:

- 수익화와 동기화가 깔끔하다.
- iOS/Android 간 경험이 일관된다.
- 가격 추적, AI 재분류 같은 Pro 기능 확장이 쉽다.

단점:

- 첫 출시가 늦어진다.
- 개인용 앱으로 쓰기 전에 가입 장벽이 생긴다.
- 운영 비용이 먼저 발생한다.

### Approach C: 저장 봇 + 모바일 뷰어

요약:
카카오톡/텔레그램/디스코드 같은 채널에 링크를 보내면 서버가 정리하고, 모바일 앱은 검색/확인 뷰어 역할을 한다.

장점:

- 저장 입력이 매우 빠르다.
- "나에게 보내기" 습관을 그대로 흡수할 수 있다.
- 향후 팀/가족 공유로 확장할 수 있다.

단점:

- iOS/Android 앱의 기본 경험이 약해진다.
- 외부 플랫폼 의존성이 커진다.
- 앱스토어용 독립 앱 가치가 흐려질 수 있다.

### 추천

MVP는 **Approach A: 로컬 우선 개인 저장소**로 시작한다.

단, 데이터 모델은 처음부터 계정/동기화/구독을 붙일 수 있게 설계한다. 첫 버전의 성공 기준은 "내가 실제로 매일 저장하고 다시 찾는다"이지, 완성형 SaaS 구조가 아니다.

## 9. 정보 구조

### 화면 구성

- 홈: 빠른 추가, 최근 저장 항목, 검색창, 필터 칩
- 저장함: 전체 활성 항목과 아카이브 포함 토글
- 상세: 출처 미리보기, AI 필드, 사용자 메모, 신뢰도, 아카이브/삭제

## 11. AI 추출 설계

### 원칙

- AI 결과는 초안이며, 사용자가 언제든 수정할 수 있다.
- 신뢰도가 낮은 필드는 "확인 필요"로 표시한다.
- 가격은 자동 추출 시점의 관측값으로 표시하고, 실시간 최저가처럼 표현하지 않는다.
- 원본 링크와 추출 시점을 함께 저장한다.

### 파이프라인

1. 입력 수집: 이미지 또는 URL
2. URL 타입 판별
3. URL 안전 게이트
4. 메타데이터 사전 수집
   - YouTube: 공식 Data API로 가능한 메타데이터 수집
   - 일반 URL: oEmbed 또는 Open Graph 메타태그 사용
   - 쇼핑몰/레시피: 페이지 메타태그와 본문 일부 사용
   - Instagram: 공식/허용된 임베드 메타데이터가 없으면 사용자가 제공한 링크와 썸네일 수준으로 제한
5. AI 구조화
6. 스키마 검증
7. 사용자 확인 화면
8. 저장

### 스키마 버전

AI 출력은 `schema_version`을 둔다. 예: `picklog_item_v1`.

모델이 바뀌어도 기존 저장 데이터는 깨지지 않아야 한다.

### URL 안전 게이트

서버 메타데이터 수집기는 다음 조건을 만족해야 한다.

- `http`와 `https`만 허용한다.
- `localhost`, 사설 IP, link-local IP, cloud metadata IP, loopback 주소를 차단한다.
- DNS 조회 후 IP 대역을 검증하고, redirect 이후의 최종 URL도 다시 검증한다.
- redirect 횟수, 응답 크기, 다운로드 시간, 본문 일부 추출 크기를 제한한다.
- 허용된 콘텐츠 타입만 처리하고, HTML은 스크립트 실행 없이 텍스트/메타태그만 파싱한다.
- 페이지 본문은 AI 지시문으로 신뢰하지 않는다. 본문은 데이터로만 취급하고, 시스템 프롬프트와 스키마를 덮어쓸 수 없다.

### AI 품질 평가

첫 30개 실제 저장 데이터셋을 AI eval 기준으로 사용한다.

- 필수 필드: `title`, `category`, `source_type`, `use_case`, `tags`
- 쇼핑 필드: `seller`, `brand`, `product_name`, `price`, `currency`
- 레시피 필드: `dish_name`, `ingredients`, `cooking_time_minutes`
- 영상 필드: `creator`, `platform`, `video_title`, `why_saved`

Alpha 통과 기준은 필수 필드 허용 정확도 80% 이상, 불확실한 가격/판매처의 "확인 필요" 표시 100%이다.

## 12. 정책과 플랫폼 제약

### iOS

- 사용자의 사진, 링크, 메모는 개인 데이터이므로 제3자 AI 처리 사실과 목적을 명확히 고지해야 한다.
- Pro 기능은 디지털 기능 잠금 해제이므로 App Store 결제 정책에 맞춰 인앱 구매 또는 구독을 우선 고려한다.
- 사용자가 저장한 외부 콘텐츠가 앱 내에서 보이는 경우, 부적절한 콘텐츠 신고/삭제/차단 흐름을 최소한 갖춘다.

### Android

- AI가 생성하거나 요약한 내용에 대해 사용자 신고/피드백 수단을 제공한다.
- 구독은 지속적 가치를 제공해야 한다. 단순 일회성 AI 추출 묶음만 구독으로 파는 것은 피한다.

### 외부 플랫폼

- YouTube는 공식 API로 제목, 설명, 썸네일, 채널명 등 영상 리소스 메타데이터를 얻는 방향이 안전하다.
- oEmbed는 URL을 직접 파싱하지 않고 임베드 가능한 대표 정보와 썸네일을 가져오는 표준 수단으로 사용할 수 있다.
- Instagram, 쇼핑몰, 레시피 사이트는 플랫폼별 접근 제한이 다르므로 무리한 크롤링보다 사용자 제공 링크, 공개 메타태그, 공식 API, 수동 보정 UX를 조합한다.

- AI 추출 로그에서 불필요한 원본 본문 제거

### 삭제와 보존 정책

| 데이터 | 기본 보존 | 휴지통 30일 내 복원 | 영구 삭제 후 처리 |
| --- | --- | --- | --- |
| `items` | 계속 보존 | 복원 가능 | 제목/메모/메타데이터 삭제, 동기화 tombstone만 기한까지 유지 |
| `attachments` | 사용자가 저장한 원본 | 복원 가능 | 로컬/서버 이미지 삭제 |
| `extractions.input_snapshot` | 최소 입력 정보 | 복원 가능 | 삭제 또는 URL만 익명화 |
| `extractions.raw_output_json` | 7일 | 복원과 무관 | 7일 후 자동 삭제 |
| `extractions.normalized_output_json` | 항목과 함께 보존 | 복원 가능 | 항목과 함께 삭제 |
| `price_observations` | 항목과 함께 보존 | 복원 가능 | 항목과 함께 삭제 |
| `sync_queue` / tombstone | 동기화 완료까지 | 복원 가능 | `tombstone_until` 이후 삭제 |

계정 삭제는 모든 서버 데이터, 결제 권한 연결 정보, 저장 이미지, 추출 로그를 삭제한다. 결제 영수증처럼 법적/회계상 보존이 필요한 데이터는 사용자 콘텐츠와 분리해 최소 범위로 보존한다.

### API와 모델 사용

- API 키와 모델 호출은 서버에서만 처리한다.
- 사용자의 사진, 링크 본문, 메모를 모델 학습에 사용하는지 여부를 설정/개인정보 고지에 명시한다.
- AI 제공자에 전송되는 데이터 범위와 보존 기간을 사용자가 이해할 수 있게 표시한다.
- URL 수집기와 AI 추출기는 요청 단위 로깅을 남기되, 원문 본문과 이미지는 기본적으로 오래 보존하지 않는다.

## 18. 구현 권장안

초기 기술 선택:

- 앱: React Native + Expo
- 로컬 DB: SQLite
- 서버: Supabase 또는 Firebase
- AI 추출 API: 서버 함수에서 호출
- 이미지 저장: Supabase Storage 또는 Firebase Storage
- 결제: iOS StoreKit, Google Play Billing, 또는 RevenueCat

이유:

- 개인 프로젝트에서 iOS/Android를 동시에 만들 수 있다.
- 공유 시트, 카메라, 이미지 선택, 로컬 DB, 푸시, 결제까지 생태계가 충분하다.
- 나중에 웹 앱으로 확장하기 전에 모바일 핵심 습관을 검증할 수 있다.

주의:

- 공유 시트 수신은 단순 공유보다 까다롭다. Expo Sharing은 수신 설정을 제공하지만 iOS Share Extension과 Android share intent 설정이 필요하고, 수신 흐름은 실제 기기 검증이 필요하다.
- 따라서 Alpha에서는 앱 안의 링크 붙여넣기와 사진 선택을 먼저 완성하고, Beta에서 공유 시트를 붙인다.

## 19. 수용 기준

MVP는 다음 조건을 만족하면 완료로 본다.

- 사용자가 사진을 저장하고 AI 메타데이터를 확인/수정/저장할 수 있다.
- 사용자가 링크를 저장하고 AI 메타데이터를 확인/수정/저장할 수 있다.
- URL 메타데이터 수집기는 SSRF 방어 테스트를 통과한다.
- 저장 항목은 리스트와 상세 화면에서 확인할 수 있다.
- 이름, 카테고리, 태그, 판매처, 가격으로 검색할 수 있다.
- 항목을 아카이브하고 복원할 수 있다.
- 항목을 삭제하고 휴지통에서 복원할 수 있다.
- 영구 삭제 시 항목, 첨부, 추출 로그, 가격 기록이 보존 정책대로 삭제 또는 익명화된다.
- AI 추출 실패 시 재시도 또는 수동 입력으로 저장을 완료할 수 있다.
- 30개 Alpha 데이터셋에서 필수 필드 허용 정확도 80% 이상을 달성한다.
- 저장 항목의 70% 이상을 예상 검색어로 다시 찾을 수 있다.
- 개인정보와 AI 처리 고지가 앱 안에 존재한다.

각 항목에 대해 다음을 기록한다.

- 입력 타입: 사진, 쇼핑몰 링크, 쇼츠 링크, 레시피 링크
- 왜 저장했는지
- 나중에 어떤 단어로 찾을 것 같은지
- AI가 자동으로 뽑아주면 편할 필드
- 각 필드의 기대값: 제목, 카테고리, 출처 타입, 사용 목적, 태그, 가격/판매처 또는 레시피/영상 핵심 필드
- AI가 틀려도 괜찮은 범위와 반드시 맞아야 하는 범위

이 30개가 Picklog의 첫 번째 테스트 데이터셋이다. 이 데이터셋에서 잘 작동하면 앱의 첫 버전은 충분히 만들 가치가 있다.
